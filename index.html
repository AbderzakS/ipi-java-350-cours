La qualité logicielle

 Une crise du logiciel est survenue dans les années 60-70.
  • Des délais de livraison non tenus
  • Des dépassements de 50% à 70% des coûts prévisionnels
  • Des anomalies techniques
  • Des décalages avec les besoins du client
  • Des difficultés à réaliser des évolutions
  • Des difficultés à réutiliser le logiciel dans d'autres contextes de développement
  • ...

Les idées reçues

 La qualité logicielle véhicule un certain nombre d'idées reçues...
  • Tester coûte cher !
  • Tester est une perte de temps !
  • Si ça ne marche pas, soit c'est le client qui s'est mal exprimé, soit c'est l'utilisateur qui ne sait pas se servir du logiciel !
  • Une fois livré, les problèmes ne sont plus de ma responsabilité...
  • On ne peut tester un logiciel que lorsqu'il est codé !
  • Ce sont les testeurs qui doivent tester !
  • Les tests doivent être automatiques !
  • Plus il y a de tests mieux c'est !
  • Tester permet de trouver tous les bugs !
  • Si tous mes tests passent, mon logiciel est de qualité !

Le développement itératif

 Peu importe la méthode choisie (enfin presque...), l'important est d'adopter une approche itérative dans les phases de développement.
 Le but étant d'avoir un produit fonctionnel (même partiel) à montrer au client régulièrement et ainsi éviter l'effet tunnel.

Fiabilité

 La fiabilité d'un code et sa capacité à s'exécuter correctement dans les cas nominaux comme dans les cas aux limites, et à produire le résultat escompté.
  • Tests unitaires
  • Tests d'intégration
  • Métriques
 C'est le niveau le plus simple à mettre en place.

Les tests unitaires

 Les tests unitaires ont pour but de tester un partie ou une portion de code de manière unitaire et indépendante.
  • Un bon test unitaire doit être...
  • unitaire
  • indépendant
  • simple
  • reproductible
  • rapide
  • clair
 Idéalement, ils sont codés avant l'implémentation de la fonctionnalité, on appelle cela le TDD pour Test Driven Development.

Les tests d'intégration

 Les tests d'intégration ont pour but de tester le bon fonctionnement de l'assemblage de modules (précédemment testés de manière unitaire) en un composant de plus haut niveau.
  • Un bon test d'intégration doit être...
  • indépendant
  • rapide
  • pertinent
 On ne doit pas re-tester les résultats produits par les sous-modules mais bien leur interaction !

Métriques

 Les métriques permettant de s'assurer de la fiabilité d'un logiciel sont les suivantes (liste non exhaustive) :
  • Résultats des tests unitaires et d'intégration (% de tests OK)
  • Couverture de code (%)
  • Couverture de code avec mutation (%)
  • Complexité (application, classe, méthode)
  • Lignes de code en commentaire
  Attention, viser systématiquement 100% de couverture de code n'est pas efficient ! Il est également nécessaire de bien paramétrer son gestionnaire de qualité afin d'éviter les faux positifs.

Capacité fonctionnelle

 La capacité fonctionnelle d'un logiciel est le degré de conformité du logiciel aux besoins du client.
  • Tests d'acceptation
  • Tests d'interface
  • Tests manuels
  • Métriques
 Il est important de connaître où est la valeur dans le logiciel pour le client, et dans la mesure du possible, commencer le développement des fonctionnalités ayant le plus de valeur.

Les tests d'acceptation

 Les tests d'acceptation vérifient la conformité fonctionnelle de l'application, c'est-à-dire sa capacité à produire les résultats décris dans les spécifications en fonction de scénarios fonctionnels.
  • Un bon test d'acceptation doit être...
  • non technique
  • uniquement fonctionnel
  • non équivoque
  • écrit par le client
  • écrit avant le développement
  • automatisé
 Les tests d'acceptation peuvent être considérés comme des spécifications. Lorsqu'ils sont fournis avant le début des développement, on parle d'ATDD ou Acceptance Test Driven Development.

Les tests d'interface

 Les tests d'interface permettent d'assurer le bon fonctionnement d'une interface graphique en simulant un comportement utilisateur. Du fait du caractère évolutif d'une interface graphique, ces tests sont souvent réalisés à la fin de développement.
  • Un bon test d'interface doit être...
  • automatisé
  • déployé sur plusieurs plateformes
  • indispensable
  • ... Y a-t-il vraiment de bons tests d'interface ?
 Les tests d'interface sont souvent difficiles à développer, automatiser et maintenir. Ils doivent donc être mis en place que pour les éléments non testables autrement (positionnement d'éléments, affichage de message d'erreur, déclenchement des actions avec les éléments de l'interface...).

Les tests manuels

 Les tests manuels ne doivent pas pour autant être oubliés. Ils sont parfois plus pertinents que des tests d'interface difficilement maintenables car plus intelligents.
  • Un bon test manuel doit être...
  • automatisé
  • scénarisé
  • pas trop long...
 Les tests manuels sont évidemment coûteux mais peuvent avoir une vraie valeur ajoutée dans certains projets. Les scénarios testés sont généralement ceux déroulés lors de la phase de recette.

Métriques

 Il n'y a pas vraiment de métriques permettant d'assurer la capacité fonctionnelle. On peut néanmoins citer :
  • Résultats des tests d'acceptation (% de tests OK)
  • Résultats des tests d'interface (% de tests OK)
  • Résultats des tests manuels (% de tests OK)
  • Satisfaction client (largeur de sourire ?)
  Il est très difficile d'impliquer le client dans cette démarche car cela engage sa responsabilité et implique qu'il connaît exactement son besoin...

Performances

 Les performances d'une application dépendent de son type. On ne mesure pas de la même manière les performances d'un site web que d'un batch d'intégration de fichier par exemple.
  • Tests de charge
  • Métriques
 Négliger les performances d'une application peut parfois coûter très cher lorsqu'il faut les améliorer une fois l'application développée et livrée...

Les tests de charge

 Il est important de commencer les tests de charge dès le début des développements car généralement, c'est au fur et à mesure des développements que les performances se déteriorent.
  • Un bon test de charge doit être...
  • pertinent
  • surveillé
  • complet
  • réaliste
 Le bon équilibre doit être trouvé car lorsqu'on améliore les performances d'un côté, c'est souvent au détriment d'un autre module.

Métriques

 Les métriques permettant de s'assurer des performances d'un logiciel sont les suivantes (liste non exhaustive) :
  • Web : Temps de chargement des pages (durée)
  • Web : Temps des recherches (durée)
  • Batch : Nombre de lectures/écritures (nombre par seconde)
  • Application : Temps de chargement
  • Temps d'exécution des tests
  • Evolution des tests de charge
 Il faut arriver à identifier les points faibles de l'application (souvent la base de données mais pas que...) car parfois la solution n'est pas logicielle (augmentation mémoire...).

Maintenabilité

 la maintenabilité correspond à la capacité d'une application livrée à être maintenue (maintenance corrective et évolutive) en maîtrisant les coûts, délais et son intégrité.
  • Journalisation
  • Documentation
  • Développements spécifiques
  • Livraisons
  • Métriques
 On s'occupe souvent de la maintenabilité à la fin de projet, lorsque la phase d'exploitation approche. C'est en général moins coûteux de garder cet élément à l'esprit au fur et à mesure des développements.

Journalisation

 La journalisation permet de consigner les événements survenus dans l'application. Ces événements sont généralement consignés dans des fichiers (avec des rotations) ou dans une base de données.
  • Actions utilisateur (CRUD)
  • Compte-rendu de traitements automatiques
  • Informations sur le client (OS, navigateur...)
  • Seuils techniques ou fonctionnels atteints
  • Erreurs techniques ou fonctionnelles
  Il existe des outils pour exploiter de manière efficace ces logs qui peuvent être difficile à consulter.

Documentation

 La documentation du code est très importante pour assurer la lisibilité et la maintenabilité d'une application 
/*
 *  Première classe d'exemple
 *  pour le cours de test
 */
public class HelloWorld
{
    /**
     * Point d'entrée du programme
     * 
     * @param args Tableau des arguments passés au programme
    */
    public static void main(String[] args)
    {
        //Affiche Hello World dans la console
        System.out.println("Hello World");
    }
}
 Cela facilite la lecture du code pour les futurs développeurs qui modifieront le code.

Documentation projet

 Une documentation technique pour le projet est un bon moyen d'améliorer le temps de montée en compétences des nouveaux arrivants.
  • MkDocks
  • Sharepoint
  • Wiki
 C'est aussi un bon endroit pour regrouper les instructions de livraison, les caractéristiques des différents environnements...

Développements spécifiques

 Il est parfois utile de développer spécifiquement des modules pour faciliter la maintenabilité et l'exploitabilité d'une application.
  • Interface de monitoring de traitements
  • Pages web d'administration (CRUD)
  • Suivi des interventions
  • Suivi des montées de version
  • Automatisation de tâches d'exploitation
  • ...
 Ce genre de développement pouvant être coûteux, il faut s'assurer de leur ROI.

Métriques

 Les métriques permettant de s'assurer de la maintenabilité d'un logiciel sont les suivantes (liste non exhaustive) :
  • Nombre de lignes de commentaires et % par rapport au code
  • Code smells (nombre)
  • Dette technique (durée)
  • Code dupliqué (% et nombre de lignes)
 Ces métriques sont à surveiller tout au long de la phase de développement.
